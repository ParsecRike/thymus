Fragment Routing and Transition Control

DOM routing attributes

There are times where it's desirable to load fragment views and/or models after some action has taken place. In this case, one of the HTTP Methods/verbs can be used whenever one or more DOM events are received. thymus.js tries to conform to RESTful conceptual design patterns. So naturally, wiring fragmented model/views into thymus.js requires the use of guidelines set forth by the established HTTP methods. Determining how model data and/or views are transfered to a page is determined by the type of HTTP method verb used on one of the corresponding data-thx-*VERB* DOM routing attributes; where *VERB* would be replaced with one of the following HTTP method verbs (e.g. data-thx-get ):

  • get
  • post
  • put
  • delete

From this point on we will refer to data-thx-*VERB* as a DOM routing attribute to reference the entire suite of supported thymus.js event driven HTTP method verb attributes. DOM routing attributes should reside on the node where the DOM event will be triggered. The semantics for DOM routing attribute are separated into multiple facets that control behavior:

event ‡
data-thx-*VERB*
The DOM routing attribute's event parameter indicates a DOM event that will trigger the fragment transition from/to the client. While most of the DOM events will directly equate to the normal $.on(event, ...); registrations, there is one special case that thymus.js will pickup regardless of the element that the DOM routing attribute is defined on. A load event will cause thymus.js to automatically process the fragment when the page first loads provided that loading has not been deferred. This type of loading is very similar to the short-hand version, but allows for additional siphons to be declared to permit more control over how fragments are handled.
parameter siphon †
data-thx-*VERB*-params
The DOM routing attribute's parameter siphon is a JQuery selector that will capture parameters that will be sent along with the routing path. See the fragment siphons section for more details on how parameter siphons use JQuery selectors.
routing path †
data-thx-*VERB*-path
The DOM routing attribute's path should point to the location where the fragment will be captured (e.g. path/to/my/frag). See the path resolution section for more details on how paths are resolved.
result siphon †
data-thx-*VERB*-result
The DOM routing attribute's result siphon is typically a JQuery selector, but can also be a function that will handle the result. When a fragment result contains view content or is XML model content the result siphon will be used to select the portion of the fragment result that will be used in the DOM. When the fragment result contains model content that is not XML the result siphon can either be omitted to use the entire fragment result or contain a function that will manually handle selecting the portion of the fragment that will be included in the DOM (e.g. using a 3rd party library to handle the selection- like json:select()). See the fragment siphons section for more details on how result siphons use JQuery selectors.
destination siphon †
data-thx-*VERB*-dest
The DOM routing attribute's destination siphon is a JQuery selector that will be used to define destination nodes within the DOM where the result siphoned fragment results will be placed. How fragment results are placed within the destination is determined by the DOM type. See the fragment siphons section for more details on how destination siphons use JQuery selectors.
DOM type †
data-thx-*VERB*-type
The DOM routing attribute's DOM type designates how a fragment result will be placed within the DOM. Valid values are:
  • include - will append the fragment result within the node where the DOM routing attribute is defined
  • replace - will replace the the node where the DOM routing attribute is defined with the fragment result
  • update - will first remove any pre-existing destination siphons before including the fragment result (if there is no corresponding destination siphon any content within the node where the DOM type is defined will be replaced with new fragment results as incoming fragment data is recieved)
‡ indicates that any updates made to the attribute require a call to re-register the DOM action before any changes will take effect
† indicates that any updates made to the attribute will automatically take effect the next time the DOM event is triggered

Simple Navigation

Navigating to other pages can be as simple as providing a normal link that contains an href (see attribute path resolution techniques for more details). A DOM event can also be used by just providing a DOM routing attribute with just an event name, an optional parameter siphon and a corresponding routing path: 

<button data-thx-get="click" data-thx-get-path="path/to/other/page" type="button">Sign in</button>
Working example that will navigate to an example sign in page

Self-targeting Asynchronous Transfers

A self-targeting asynchronous transfer is a DOM routing attribute that has a value that defines a event, an optional parameter siphon, a routing path and an optional result siphon, but does not define a destination siphon. In this case the destination siphon will default to the node where the DOM routing attribute is defined. Consider the following: 

<!-- assume a fragment resides at path/to/user that contains -->
<div class="user-detail">
	...
</div>
<!-- assume a page contains the following -->
<div>
	<h1>User Data:</h1>
	<div data-thx-get="click" data-thx-get-path="path/to/user" data-thx-get-result=".user-detail">
		<strong>Click to load next user...</strong>
	</div>
</div>
<!-- when the div is clicked and the fragment is loaded and will result in -->
<div>
	<h1>User Data:</h1>
	<div data-thx-get="click" data-thx-get-path="path/to/user" data-thx-get-result=".user-detail">
		<strong>Click to load next user...</strong>
		<div class="user-detail">
			...
		</div>
	</div>
</div>
<!-- another click on the div would result in -->
<div>
	<h1>User Data:</h1>
	<div data-thx-get="click" data-thx-get-path="path/to/user" data-thx-get-result=".user-detail">
		<strong>Click to load next user...</strong>
		<div class="user-detail">
			...
		</div>
		<div class="user-detail">
			...
		</div>
	</div>
</div>
<!-- and so on... -->

<!-- assume a fragment resides at path/to/user that contains -->

		Load User
	</button>
</div>
<!-- when the div is clicked and the fragment is loaded and will result in -->
<div>

		...
	</div>

Explicit Destination Asynchronous Transfers

The most granular of the DOM event driven behaviors is one that defines a destination siphon. The destination siphont tells thymus.js where the returned fragment should be placed. Following the previous trend of examples... 

<!-- assume a fragment resides at path/to/user that contains -->
<div class="user-detail">
	...
</div>
<!-- assume a page contains the following -->
<div>
	<button type="button"
		data-thx-get="click" data-thx-get-path="path/to/user" data-thx-get-result=".user-detail" 
		data-thx-get-dest="#userDest">Load User</button>
	<div id="userDest">
		<strong>Users:</strong>
	</div>
</div>
<!-- when the button is clicked and the fragment is loaded and will result in -->
<div>
	<button type="button"
		data-thx-get="click" data-thx-get-path="path/to/user" data-thx-get-result=".user-detail" 
		data-thx-get-dest="#userDest">Load User</button>
	<div id="userDest">
		<strong>Users:</strong>
		<div class="user-detail">
			...
		</div>
	</div>
</div>
<!-- another click on the button would result in -->
<div>
	<h1>User Data:</h1>
	<button type="button" 
		data-thx-get="click" data-thx-get-path="path/to/user" data-thx-get-result=".user-detail" 
		data-thx-get-dest="#userDest">Load User</button>
	<div id="userDest">
		<strong>Users:</strong>
		<div class="user-detail">
			...
		</div>
		<div class="user-detail">
			...
		</div>
	</div>
</div>
<!-- and so on... -->

Auto Loading Fragments (short-hand)

A short-hand routing attribute can also be used which is equivalent to a DOM routing attribute with a load event and a separate routing path attribute. Instead of two attributes, the short-handed version combines the routing path and result siphon into one attribute separated by a "::" (the separator can be overridden using the data-thx-separator attribute on the thymus.js script definition or programmatically by passing the appropriate option parameter). It also infers the DOM type based upon the short-hand attribute used. The short-handed version can be used as either a replacement or inclusion anywhere a data-thx-replace or data-thx-include routing attribute is found throughout the document (including within a fragment itself). It's use is limited to the routing path and result siphon defined within it and will not use any other DOM routing attributes that may be defined on the node. Examples can be found under the fragment types section.

The following example demonstrates how to automatically load a fragment using a data-thx-replace attribute: 

<!-- assume a resource resides at "frags/nav/nav" -->
<!-- that contains the following -->
<div class="topnav">
	...
</div>
<!-- and the current page contains -->
<body>
	<div data-thx-replace="frags/nav/nav :: .topnav"></div>
</body>
<!-- would result in -->
<body>
	<div class="topnav">
		...
	</div>
</body>

Of course, we could have also used a data-thx-include instead of data-thx-replace which would have appended the fragment rather than replacing the placeholder: 

<!-- assume a resource resides at "frags/nav/nav" -->
<!-- that contains the following -->
<div class="topnav">
	...
</div>
<!-- and the current page contains -->
<body>
	<div data-thx-include="frags/nav/nav :: .topnav">
		<span>Top Nav</span>
	</div>
</body>
<!-- would result in -->
<body>
	<div data-thx-include="frags/nav/nav :: .topnav">
		<span>Top Nav</span>
		<div class="topnav">
			...
		</div>
	</div>
</body>

Templates can also refer to fragments that exist in the same page that the routing attribute resides in by using this or omitting the routing attribute's path altogether (e.g. data-thx-include="::.my-inline-frag" ). Only use this approach when the fragment result siphon is not referencing nodes that reside in other fragments. 

<!-- a page with -->
<div>
	<div data-thx-include="this :: .my-inline-frag"></div>
</div>
<div>
	<div class="my-inline-frag">
		...
	</div>
</div>
<!-- would produce -->
<div>
	<div data-thx-include="this :: .my-inline-frag">
		...
		<div class="my-inline-frag">
			...
		</div>
	</div>
</div>
<div>
	<div class="my-inline-frag">
		...
	</div>
</div>

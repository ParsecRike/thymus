Path Resolution

Fragment Path Resolution

Because thymus.js runs on the client it's not inherently application context aware. Therefore, thymus.js uses a configurable data-thx-context-path attribute on the thymus.js script tag. It's value establishes the path to the root context of the application that will be resolved along with the fragment's path siphon. path siphons can point to resources with different path levels. So, it's important that the data-thx-context-path synchronizes with the resources defined within the path siphon. For example, a fragment that resides in example.com/my-app/frags/nav.htm may also be included in example.com/my-app/index.htm as well as example.com/my-app/user/info.htm. If the routing attribute is defined something like: 

<div id="topnav">
	...
</div>

And both example.com/my-app/index.htm and example.com/my-app/user/info.htm contain a corresponding routing attribute: 

<div data-thx-include="frags/nav/nav :: #topnav">
	...
</div>

Then the thymus.js script definitions would be as follows: 

<head>
<meta charset="utf-8" />
<!-- example.com/my-app/index.htm -->
<script src="js/thymus.js" id="thymus" data-thx-context-path="."></script>
</head>

<head>
<meta charset="utf-8" />
<!-- example.com/my-app/user/info.htm -->
<script src="../js/thymus.js" id="thymus"data-thx-context-path="../" ></script>
</head>

Notice that in both resources the data-thx-context-path points to example.com/my-app with my-app being the root context (example.com/my-app/index.htm uses "." which infers the current path; which would be example.com/my-app/ while example.com/my-app/user/info.htm uses "../" which infers one directory up; which would be example.com/my-app/). This vastly simplifies path siphon resolution (especially for nested fragments) and ensures that all path siphons are relative to the root context path of the application. It also permits the use of the same context path to be used within multiple path siphons; regardless of where the page that defines it resides.

Fragment Path Substitutions

When URLs are dependent upon user input static URLs will not suffice. In this case a path substitution can be made using a JQuery selector by siphoning values from other DOM elements.

Fragment File Extensions

File extensions in path siphons are optional (and sometimes undesirable). That's why thymus.js, by default, will use the URL defined in a path siphon per verbatim unless a data-thx-frag-extension is defined. If a data-thx-frag-extension attribute is defined on the thymus.js script it will use the data-thx-frag-extension value as the file extension and will append it to any path siphon URL that does not already have a file extension (should typically be prefixed with a period). Any path siphon URL that has it's own file extension will not get the data-thx-frag-extension appended to it's path siphon. A special value of inherit can be used if fragments should inherit the parent page file extension (if any). Here are a few variations: 

<!-- assume the following script -->
<script src="../js/thymus.js" id="thymus"
	data-thx-context-path="." data-thx-frag-extension=".htm"></script>
<!-- and the following includes/replacements -->
<div data-thx-include="/frags/person :: #data"></div>
<div data-thx-include="/frags/person/ :: #data"></div>
<div data-thx-include="/frags/person.xml :: #data"></div>
<!-- would result in -->
<div data-thx-include="/frags/person.htm :: #data"></div>
<div data-thx-include="/frags/person/ :: #data"></div>
<div data-thx-include="/frags/person.xml :: #data"></div>

<!-- assume the following script is loaded into a page at path/to/page.html -->
<script src="../js/thymus.js" id="thymus"
	data-thx-context-path="." data-thx-frag-extension="inherit"></script>
<!-- and the following includes/replacements -->
<div data-thx-include="/frags/person :: #data"></div>
<div data-thx-include="/frags/person/ :: #data"></div>
<div data-thx-include="/frags/person.xml :: #data"></div>
<!-- would result in -->
<div data-thx-include="/frags/person.html :: #data"></div>
<div data-thx-include="/frags/person/ :: #data"></div>
<div data-thx-include="/frags/person.xml :: #data"></div>

<!-- assume the following script -->
<script src="../js/thymus.js" id="thymus"
	data-thx-context-path="."></script>
<!-- and the following includes/replacements -->
<div data-thx-include="/frags/person :: #data"></div>
<div data-thx-include="/frags/person/ :: #data"></div>
<div data-thx-include="/frags/person.xml :: #data"></div>
<!-- would result in -->
<div data-thx-include="/frags/person :: #data"></div>
<div data-thx-include="/frags/person/ :: #data"></div>
<div data-thx-include="/frags/person.xml :: #data"></div>

Attribute Path Resolution

As with Fragment Path Resolution, relative URLs within some standard URL related attributes are resolved to their absolute URL counterparts relative to the root context of the application as defined by the data-thx-context-path on the thymus.js script . Although automatically resolving URL related attributes is configurable, by default, the following attributes are resolved:

  • action
  • archive
  • background
  • cite
  • classid
  • codebase
  • data
  • dynsrc
  • formaction
  • href
  • icon
  • longdesc
  • lowsrc
  • manifest
  • poster
  • profile
  • src
  • usemap

Any of the resolved attributes that contain URLs that begin with an IANA registered scheme or begin with a / will NOT be updated by thymus.js. When any of the resolved attributes within a fragment begin with a protocol-relative URL (i.e. //) and the page is being called using a file URI scheme the URL will default to http: in order to preserve external URL integrity. Here are a few examples: 

<!-- a page at: http://example.com/my-app/login.htm -->
<!-- with a data-thx-context-path="." that contains a fragment with a -->
<a href="index.htm">...</a>
<!-- will be result in -->
<a href="http://example.com/my-app/index.htm"></a>

<!-- a page at: http://example.com/my-app/user/info.htm -->
<!-- with a data-thx-context-path="../" that contains a fragment with a -->
<a href="index.htm">...</a>
<!-- will be result in -->
<a href="http://example.com/my-app/index.htm"></a>

<!-- a page at: http://example.com/my-app/index.htm -->
<!-- with a data-thx-context-path="." that contains a fragment with a -->
<a href="user/info.htm">...</a>
<!-- will be result in -->
<a href="http://example.com/my-app/user/info.htm"></a>

<!-- a page at: file://example.com/my-app/index.htm (notice the file: scheme) -->
<!-- with a data-thx-context-path="." that contains a fragment with a -->
<link
	href="//code.jquery.com/ui/jquery-ui-git.css"
	rel="stylesheet" />
<!-- will be result in -->
<link
	href="http://code.jquery.com/ui/jquery-ui-git.css"
	rel="stylesheet" />

Siphon Resolution

A siphon in a most practical sense is just one or more JQuery selectors with an optional insertion/extraction directives, but can also be a function that will handle the extraction and/or placement of data within the DOM. JQuery selectors can be used with or without an appended -> directive:

  • text - will use JQuery.text() to insert/extract value(s) returned from the JQuery selected node(s)
  • a name of an attribute - will use JQuery.attr() to insert/extract value(s) returned from the JQuery selected node(s)
  • no directive (default) - will use values from JQuery.serializeArray() or value(s) from JQuery.val() (when applicable) to insert/extract value(s) returned from the JQuery selected node(s)

Lets consider a path siphon where more than one JQuery selector can be used. In this case each JQuery selector and corresponding extraction directive are wrapped within a @{} . 

<button data-thx-get="click" 
	data-thx-path="frags/first/@{#secondPath->text}/sixth">Get Something</button>
<div id="secondPath">@{#second}/@{#thirdFourthPath :input}/@{.fifth-path->data-mypath}</div>
<select id="second" name="secondPath">
    <option>second</option>
    <option>2nd</option>
</select>
<div id="thirdFourthPath">
    <input type="text" name="someInput" value="third" />
    <input type="text" name="someOtherInput" value="fourth" />
</div>
<div class="fifth-path" data-mypath="fifth">

The preceeding path siphon would get evaluated when the button is clicked. When the first URL siphon is discovered within the wrapped @{} it will capture the node(s) returned by the JQuery #secondPath selector. The -> defines a text directive that will use the JQuery.text() function to extract the value. Once the path segment is captured from the #secondPath discovery occurs using the same extraction methods on seach of the @{} segments found in @{#second}/@{#thirdFourthPath :input}/@{.fifth-path->data-mypath} . The #second JQuery selector has no -> directive so it uses the default JQuery.val(). Next, the #thirdFourthPath :input JQuery selector is used to capture node(s) and JQuery.serializeArray() is called on the returned node(s) because there is no -> directive and there is no JQuery.val() on div s. Each of the serialized array values are captured and inserted into the path separated by a / . The final .fifth-path JQuery selector is used to capture the node(s) where the -> directive for the attribute data-mypath is evaluated and inserted into the path. The siphoned path then becomes frags/first/second/third/fourth/fifth/sixth .

JQuery Selector Fragment Targets

JQuery selectors can be used to identify the retrieved fragment's DOM element(s) that will be included or replaced within the DOM. For example: 

<!-- assume a resource resides at "frags/nav/user" -->
<!-- that contains the following fragment -->
<body>
	<div class="user">
		...
	</div>
</body>
<!-- and the current page contains -->
<body>
	<div data-thx-include="frags/user/user :: .user">
		<span>User</span>
	</div>
</body>
<!-- would result in -->
<body>
	<div data-thx-include="frags/user/user :: .user">
		<span>User</span>
		<div class="user">
			...
		</div>
	</div>
</body>

A data-thx-replace would not make sense if the selector returned more than one element (in which case only the last found element would be present in the replacement). However, a data-thx-include with a selector that returns more than one result would: 

<!-- assume a resource resides at "frags/user/user" -->
<!-- that contains the following fragment -->
<body>
	<ul class="users">
		<li>User 1</li>
		<li>User 2</li>
	</ul>
</body>
<!-- and the current page contains -->
<body>
	<ol data-thx-include="frags/user/user :: .users li">
	</ol>
</body>
<!-- would result in -->
<body>
	<ol data-thx-include="frags/user/user :: .users li">
		<li>User 1</li>
		<li>User 2</li>
	</ol>
</body>
Reminder! Make sure to escape any quotations that may exist in your selector 
<div data-thx-include="frags/user/user :: [class*=&quot;icon-&quot;]"></div>
